# 📉 8점대 → 3점대로 성능이 개선된 이유

## 핵심 결론
- **`target_lag` 피처 추가가 결정적 원인**

---

## 📊 기존 모델의 문제점
- 창고 운영 상태(혼잡도, 로봇 효율 등)만 활용
- **시간 흐름(시계열 패턴)을 반영하지 못함**
- 중요한 패턴:
  - “지금 지연이 심하면, 30분 후에도 심하다”
- → 가장 강력한 신호(과거 타겟 정보)가 누락된 상태

---

## 🔑 개선된 핵심 피처 (Lag Features)

```
target_lag1 = 직전 타임슬롯의 실제 지연값
target_lag2 = 2슬롯 전 지연값
target_lag3 = 3슬롯 전 지연값
```

- 시계열 데이터에서 매우 강력한 예측 변수
- **feature importance 상위 대부분을 차지했을 가능성 높음**

---

## 📈 성능 개선 요인 분석
- `target_lag` → 압도적인 영향
- 기타 피처 (기여도 낮음):
  - 시나리오 집계
  - 상대 위치
  - 앙상블 등

👉 결론:  
**다른 개선보다 `lag 피처`가 성능을 사실상 결정**

---

# ⚠️ 주의사항 (데이터 누수 관련)

## 📌 Lag 피처 생성 코드

```python
# 5번 섹션 - 타겟 Lag 피처
train[col_name] = train.groupby('scenario_id')[TARGET].shift(lag)
```

---

## ✔️ 안전한 이유
- train 데이터 내부에서만 처리
- test의 실제 타겟값을 사용하지 않음
- → **명백한 데이터 누수는 아님**

---

## ⚠️ 애매한 부분 (대회 규칙 관련)

### 규칙
> test 데이터는 어떠한 형태로도 모델 학습에 사용 불가

### 논쟁 포인트
- test 예측값을 다음 시점의 lag로 사용하는 방식
  - 학습이 아닌 **추론 과정**
  - → 직접적인 규칙 위반은 아님
- 하지만:
  - **경계선에 있는 전략 (gray area)**

---

## 🧭 권장 행동
- 대회 운영진 또는 토론 게시판에 문의하는 것이 안전

---

# 🧠 한 줄 정리
> 시계열 문제에서 **과거 타겟값(lag)** 은 가장 강력한 피처이며, 성능을 급격히 끌어올릴 수 있다. 다만, **test 데이터 활용 방식은 항상 규칙을 엄격히 확인해야 한다.**
